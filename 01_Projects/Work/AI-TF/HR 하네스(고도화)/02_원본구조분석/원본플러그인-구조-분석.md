# 원본 플러그인 구조 분석

## 이 문서의 목적
이 문서는 `ehr-harness-plugin` 레포를 신입 개발자 관점에서 읽기 쉽게 해체한 문서다.

이 문서에서 답하려는 질문은 네 가지다.

- 원본 플러그인은 어떤 층으로 나뉘어 있는가
- 각 폴더는 생성 과정에서 무슨 역할을 하는가
- 실제 하네스 생성은 어떤 순서로 진행되는가
- 연말정산 하네스를 만들 때 어디를 그대로 쓰고, 어디를 바꿔야 하는가

## 기준 레포
- [ehr-harness-plugin](https://github.com/qoxmfaktmxj/ehr-harness-plugin)
- 로컬 확인 경로: `.tmp/ehr-harness-plugin/plugins/ehr-harness/`

## 먼저 한 문장으로 구조를 잡기
이 레포는 "문서 템플릿 저장소"가 아니라, `공통 자산 + 프로필 자산 + 생성 엔진 + 검증 로직`을 함께 가진 하네스 생성 플러그인이다.

즉, 신입 기준으로는 아래처럼 보면 이해가 쉽다.

1. `profiles/shared`
공통판

2. `profiles/ehr4`, `profiles/ehr5`
버전별 특화판

3. `skills/ehr-harness/SKILL.md`
전체 생성 절차를 지휘하는 메타 스킬

4. `skills/ehr-harness/lib`
실제 감지, 분석, 병합, audit, state 관리를 수행하는 엔진

5. `.claude-plugin/plugin.json`
이 플러그인을 배포 가능한 단위로 묶는 메타데이터

## 최상위 구조

```text
plugins/ehr-harness/
├── .claude-plugin/
├── profiles/
│   ├── shared/
│   ├── ehr4/
│   └── ehr5/
├── scripts/
└── skills/
    └── ehr-harness/
        ├── SKILL.md
        └── lib/
```

## 폴더별 역할

| 위치 | 역할 | 신입 관점 해석 |
|------|------|----------------|
| `.claude-plugin/plugin.json` | 플러그인 이름, 설명, 버전 관리 | "이 플러그인을 설치 가능한 제품으로 묶는 표지" |
| `profiles/shared/` | 모든 프로젝트에 공통으로 심는 자산 | "기본 운영체제" |
| `profiles/ehr4/` | EHR4 전용 문서/스킬/에이전트 | "EHR4용 번들" |
| `profiles/ehr5/` | EHR5 전용 문서/스킬/에이전트 | "EHR5용 번들" |
| `skills/ehr-harness/SKILL.md` | 생성 절차 전체를 지휘하는 메타 스킬 | "설치 마법사 + 오케스트레이터" |
| `skills/ehr-harness/lib/` | detect/analyze/audit/merge/state 헬퍼 | "실행 엔진" |
| `scripts/` | 오프라인 보조 스크립트 | "유지보수용 도구함" |

## 구조를 읽는 올바른 순서
신입 개발자는 레포를 아래 순서로 읽는 것이 가장 덜 벅차다.

1. `plugin.json`
플러그인의 정체 확인

2. `profiles/shared`
모든 프로젝트에 공통으로 들어가는 자산 파악

3. `profiles/ehr4`, `profiles/ehr5`
버전별 차이 파악

4. `skills/ehr-harness/SKILL.md`
생성 절차 전체 흐름 파악

5. `skills/ehr-harness/lib`
세부 구현과 품질 장치 파악

## 1. `.claude-plugin/plugin.json`
가장 작지만 의미는 크다.

현재 확인된 정보:

- name: `ehr-harness`
- description: `EHR4/5 프로젝트용 AI 코딩 하네스 자동 생성기`
- version: `1.9.4`

이 파일이 말해주는 것은 단순하다.

- 이 레포는 실험 메모가 아니라 실제 설치형 플러그인이다.
- 버전이 올라가면서 생성 결과와 스키마도 진화한다.
- 따라서 구조 분석 시 문서만 볼 게 아니라 "버전이 바뀌면 무엇이 같이 바뀌는가"도 함께 봐야 한다.

## 2. `profiles/shared` 분석
이 폴더는 EHR4/EHR5 구분 없이 공통으로 프로젝트에 심는 파일들이다.

### 실제 포함 자산
- `settings.json`
- `hooks/db-read-only.sh`
- `hooks/vcs-no-commit.sh`
- `.codex/config.toml`
- `.gitignore`
- `commands/ehr/*.md.skel`

### 왜 중요한가
신입은 보통 프로필 폴더부터 보고 싶어 하지만, 실제로는 이 `shared`가 더 중요하다.

이유:

- 안전장치는 버전과 무관하게 먼저 깔려야 한다.
- Codex/Claude/Gemini 같은 실행 플랫폼 규칙도 공통층에 있어야 한다.
- `/ehr:*` 명령들도 여기서 공통으로 관리된다.

### 대표 파일 해석
#### `settings.json`
Claude의 `PreToolUse` 훅에 아래 두 스크립트를 연결한다.

- `db-read-only.sh`
- `vcs-no-commit.sh`

즉, 하네스를 깔자마자 AI의 행동 경계가 먼저 설정된다.

#### `hooks/db-read-only.sh`
DB 실행 명령에서 변경성 키워드를 차단한다.

대표 차단 대상:
- `DROP`
- `DELETE`
- `UPDATE`
- `INSERT`
- `MERGE`
- `ALTER`
- `CREATE`
- `EXEC`
- `CALL`

의미:
- 원본 플러그인은 "AI가 DB를 읽는 건 허용해도 바꾸는 건 막는다"는 철학을 기본 내장한다.

#### `.codex/config.toml`
Codex가 `AGENTS.md`와 `CLAUDE.md`를 보조 지침으로 따르도록 설정한다.

의미:
- 이 플러그인은 Claude 전용이 아니라 Codex까지 염두에 둔 멀티 플랫폼 하네스다.

#### `commands/ehr/*.md.skel`
공통 slash command 골격이다.

확인된 명령:
- `ideate`
- `plan`
- `work`
- `review`
- `compound`

의미:
- 이 레포는 "한 번 깔고 끝"이 아니라, 이후 계획/작업/리뷰/지식축적 사이클까지 포함하는 구조다.

## 3. `profiles/ehr4`, `profiles/ehr5` 분석
이 두 폴더는 버전별 특화 자산 묶음이다.

구조는 거의 같다.

```text
profiles/ehrX/
├── skeleton/
├── agents/
├── skills/
└── reference/
```

### `skeleton/`
프로젝트별 실측값을 주입받아 최종 문서가 되는 템플릿이다.

확인된 파일:
- `AGENTS.md.skel`
- `CLAUDE.md.skel`
- `README.md.skel`

의미:
- 최종 문서는 고정 복붙이 아니다.
- 시스템명, 모듈맵, 세션변수, 치명 프로시저 같은 값이 이 skeleton에 주입되어 렌더링된다.

### `agents/`
역할이 분리된 서브 에이전트 정의다.

확인된 파일:
- `screen-builder.md`
- `procedure-tracer.md`
- `release-reviewer.md`
- `db-impact-reviewer.md`

의미:
- 원본 하네스는 단순 검색 스킬만 제공하지 않는다.
- 화면 생성, 프로시저 추적, 릴리즈 검토, DB 영향도 검토처럼 역할 기반 자동화를 염두에 둔다.

### `skills/`
실제 반복 작업을 돕는 스킬 정의다.

확인된 스킬:
- `screen-builder`
- `codebase-navigator`
- `procedure-tracer`
- `db-query`
- `impact-analyzer`
- `design-guide`
- `domain-knowledge`

의미:
- 프로젝트 규칙만 알려주는 하네스가 아니라, "무엇을 어떻게 조사하고 생성할지"까지 안내한다.

### `reference/`
고정 참조 자산이다.

확인된 파일:
- `CODE_MAP.md`
- `DB_MAP.md`

의미:
- 생성 실패 시 fallback 역할도 하고,
- 프로젝트별 문서가 생기기 전의 기본 레퍼런스 역할도 한다.

## 4. `skills/ehr-harness/SKILL.md` 분석
이 파일이 사실상 플러그인의 본체다.

신입 기준으로는 이렇게 이해하면 된다.

- `profiles/*`는 재료 창고
- `SKILL.md`는 그 재료를 언제 어떻게 꺼내 쓸지 정하는 조립 설명서

### 핵심 단계
원본 파일에서 확인한 주요 단계는 아래와 같다.

1. `Step 0`
플러그인 위치 탐색

2. `Step 0.5`
`superpowers` 의존성 검증

3. `Step 0.7`
기존 하네스 상태 감지

4. `Step 1`
EHR4/EHR5 자동 감지

5. `Step 2`
프로젝트 심층 분석

6. `Step 3-PRE`
소스/출력 매핑과 diff 계산

7. `Step 3`
실제 파일 생성

8. `Step 4`
검증

9. `Step 4-I`
EHR Cycle 자산 생성

10. `Step 4-G`
`HARNESS.json` 갱신

### 이 구조가 의미하는 것
이 플러그인은 문서를 쓰기 전에 항상 아래 질문에 먼저 답한다.

- 지금 프로젝트가 신규인가, 기존인가
- EHR4인가, EHR5인가
- 현재 구조에서 어떤 값을 실측할 수 있는가
- 기존 파일을 덮어써도 되는가, 병합해야 하는가

즉, "생성"보다 "판단"이 먼저 오는 구조다.

## 5. `lib/` 분석
이 폴더는 품질과 유지보수성을 담당한다.

### 핵심 파일
- `detect.sh`
- `analyze.sh`
- `audit.sh`
- `merge.sh`
- `harness-state.sh`
- `db.sh`
- `ehr-compound.sh`

### 역할 해석
#### `detect.sh`
- 프로젝트가 EHR4인지 EHR5인지 감지
- 플러그인 적용의 첫 갈림길

#### `analyze.sh`
- 모듈, 세션변수, 프로시저, 법칙 카운트 등 실측값 수집
- 최종 문서의 내용 품질을 좌우

#### `audit.sh`
- 저장된 baseline과 현재 상태의 drift 비교
- 하네스를 "설치형"이 아니라 "운영형" 도구로 만드는 핵심

#### `merge.sh`
- 기존 파일을 무조건 덮어쓰지 않고 관리 섹션만 병합
- 사용자 편집 보존에 중요

#### `harness-state.sh`
- `HARNESS.json` 생성과 상태 관리
- 다음 업데이트와 audit의 기준점 역할

#### `db.sh`
- DB 연결/덤프/검증 전략 보조
- 프로시저/트리거 분석 정확도에 영향

#### `ehr-compound.sh`
- EHR Cycle 자산과 누적 지식 관리
- 단순 설치 이후의 지속적 진화를 담당

### 신입이 여기서 꼭 봐야 할 포인트
- 원본 플러그인이 "문서 생성기"가 아니라 "상태를 갖는 시스템"이라는 점
- 따라서 `HARNESS.json`, merge, audit를 빼고 이해하면 반쪽 이해가 된다

## 6. 실제 생성 파이프라인과 폴더 매핑

| 생성 단계 | 실제 참조 위치 | 결과물 |
|------|------|------|
| 공통 인프라 생성 | `profiles/shared/*` | `.claude/settings.json`, hooks, `.codex/config.toml`, `.gitignore` |
| 루트 문서 생성 | `profiles/ehr4|ehr5/skeleton/*` | `AGENTS.md`, `CLAUDE.md`, `README.md` |
| 스킬 생성 | `profiles/ehr4|ehr5/skills/*` | `.claude/skills/*/SKILL.md` |
| 에이전트 생성 | `profiles/ehr4|ehr5/agents/*` | `.claude/agents/*.md` |
| 플랫폼 호환 자산 | `.claude/skills/*` 복제 + shared 설정 | `.agents/skills/*`, `GEMINI.md` |
| 상태 기록 | `lib/harness-state.sh` | `.claude/HARNESS.json` |

이 표를 보면 구조가 단순해진다.

- `shared`는 공통 인프라
- `profile`은 문서/스킬/에이전트 원본
- `lib`는 그걸 프로젝트 상태에 맞게 조립하는 엔진

## 7. 이 레포가 단순 템플릿 저장소와 다른 이유
보통 템플릿 저장소는 "정해진 파일 세트를 복사"하는 데 그친다.

하지만 이 플러그인은 다르다.

### 이유 1. 자동 감지
- EHR4/EHR5를 자동 감지한다.

### 이유 2. 실측 주입
- 모듈 수, 세션 변수, 프로시저 목록 등을 문서 안에 직접 넣는다.

### 이유 3. 상태 관리
- `HARNESS.json`으로 생성 결과와 스키마 버전을 기록한다.

### 이유 4. 안전 병합
- 이미 있는 파일을 통째로 덮어쓰기보다 merge나 diff 판단을 시도한다.

### 이유 5. audit 지원
- drift를 비교하고 유지보수 경로까지 갖고 있다.

즉, 이 레포는 "한 번 만들고 끝"이 아니라 "계속 업데이트되는 하네스"를 전제로 설계되었다.

## 8. 연말정산 하네스 관점에서 무엇을 그대로 쓰고 무엇을 바꿔야 하나

### 그대로 가져가도 되는 축
- `shared` 구조 자체
- safety hooks 철학
- profile 분리 방식
- skeleton -> 실측값 치환 방식
- merge/audit/state 관리 개념

### 연말정산에 맞게 바꿔야 하는 축
- `reference` 내용
- `domain-knowledge` 내용
- `screen-builder`나 `procedure-tracer`의 도메인 예시
- `release-reviewer`의 검토 기준
- 치명 프로시저/핵심 테이블 목록
- 첫 slice에 필요한 질문 패턴과 탐색 경로

### 특히 중요한 포인트
연말정산 고도화는 `profiles/*` 문서만 손보면 끝나는 작업이 아니다.

같이 봐야 하는 것:
- 어떤 도메인 지식을 `reference`에 둘지
- 어떤 탐색 절차를 `skill`로 둘지
- 어떤 검토 관점을 `agent`에 둘지
- 어떤 분석 결과를 `HARNESS.json`이나 audit 기준으로 남길지

## 9. 신입 개발자가 이 레포를 볼 때 자주 헷갈리는 지점

### 1. `profiles/*`만 보면 된다고 착각하기 쉽다
하지만 실제 생성 순서를 결정하는 건 `SKILL.md`와 `lib/`다.

### 2. `shared`를 부가 자산으로 보기 쉽다
실제로는 안전장치와 공통 실행 환경이 다 여기에 있다.

### 3. `HARNESS.json`을 로그 파일 정도로 보기 쉽다
실제로는 업데이트와 audit의 기준점이다.

### 4. `domain-knowledge`만 고치면 도메인 확장이 끝난다고 보기 쉽다
실제로는 `reference`, `skill`, `agent`, `analysis 기준`까지 같이 맞춰야 한다.

## 10. 지금 우리 작업에 바로 연결되는 해석
지금 우리가 만드는 `HR 하네스(고도화)`는 원본 레포를 그대로 베끼는 작업이 아니라, 아래 구조를 이해한 뒤 연말정산 도메인에 맞는 별도 프로필/규칙/지식층을 설계하는 작업이다.

즉, 현재 관점에서의 해석은 이렇다.

- `shared`는 최대한 유지
- `profile`은 연말정산 도메인에 맞게 재설계
- `lib`는 원본 개념을 참고하되 필요한 부분만 가져와 단순화 가능
- 도메인 지식은 `reference + skill + agent` 세 층으로 나눠 넣어야 함

원본 플러그인의 핵심은 아래 한 줄로 정리할 수 있다.

`프로젝트를 먼저 분석하고, 그 결과를 공통 자산과 프로필 자산에 주입해, 안전하게 생성·업데이트·audit까지 수행하는 하네스 생성 시스템`

이 구조를 이해하면 다음 단계인 [[03_Gap분석/연말정산-도메인-gap-분석|연말정산 도메인 gap 분석]]에서 무엇이 비어 있는지 더 정확하게 잡을 수 있다.
