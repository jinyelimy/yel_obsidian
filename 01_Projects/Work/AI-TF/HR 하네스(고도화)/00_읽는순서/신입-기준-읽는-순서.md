# 신입 기준 읽는 순서

## 왜 이 순서로 보나
- 신입 개발자 입장에서는 "어떻게 만들었는지"보다 "그래서 프로젝트에 뭐가 생기는지"를 먼저 보는 편이 훨씬 덜 벅차다.
- 전체를 한 번에 설계하고 구현하기보다, 작은 단위 하나를 끝까지 돌려보는 방식이 이해와 검증에 유리하다.
- 그래서 이 공간은 `결과물 -> 메커니즘 -> gap -> 설계 -> 작은 구현 -> 작은 테스트 -> 릴리즈` 순서로 정리한다.

## 읽는 순서
1. [[01_결과물투어/하네스가-프로젝트에-입혀지는-방식|하네스가 프로젝트에 입혀지는 방식]]
질문: "원본 플러그인을 쓰면 프로젝트 안에 실제로 무엇이 생기지?"

2. [[02_원본구조분석/원본플러그인-구조-분석|원본 플러그인 구조 분석]]
질문: "그 결과물이 원본 플러그인 안에서 어떤 구조로 만들어지지?"

3. [[03_Gap분석/연말정산-도메인-gap-분석|연말정산 도메인 gap 분석]]
질문: "이 원본 하네스로 연말정산 질문을 바로 처리할 수 있나, 아니면 무엇이 비어 있나?"

4. [[04_도입설계/연말정산-도입-설계|연말정산 도입 설계]]

5. [[05_첫Slice/첫-slice-구현-계획|첫 수직 slice 구현 계획]]
질문: "전체를 건드리지 않고 가장 작은 기능 하나를 어떻게 먼저 붙일까?"

6. [[06_테스트/첫-slice-테스트-전략|첫 수직 slice 테스트 전략]]
질문: "붙인 결과가 실제로 쓸 만한지 어떻게 확인하지?"

7. [[07_릴리즈/최종하네스-릴리즈|최종 하네스 릴리즈]]
질문: "이제 이 구조를 언제 최종판으로 볼 수 있지?"

## 이 순서의 핵심 원칙
- 결과를 먼저 보고 구조를 본다.
- 설계 전에 gap을 먼저 적는다.
- 구현과 테스트를 한 번에 크게 하지 않는다.
- `기능 1개 x 연도 1개`처럼 얇은 slice부터 검증한다.
