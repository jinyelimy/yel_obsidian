# 하네스가 프로젝트에 입혀지는 방식

## 한 문장으로 먼저 이해하기
`ehr-harness-plugin`은 EHR 프로젝트를 한 번 분석한 뒤, 그 결과를 `shared` 공통 자산과 `ehr4/ehr5` 프로필 자산에 주입해서 프로젝트 루트에 하네스 파일들을 생성하는 플러그인이다.

신입 관점에서는 이렇게 이해하면 된다.

- 그냥 문서를 복사해 넣는 플러그인이 아니다.
- 프로젝트를 먼저 읽고,
- 그 결과에 맞는 프로필을 고르고,
- skeleton에 실측값을 채워 넣어,
- 실제로 바로 쓸 수 있는 하네스를 프로젝트 안에 심는 방식이다.

## 실제로 어디서 들어오나
원본 레포를 보면 하네스 자산은 크게 세 군데에서 온다.

### 1. `profiles/shared`
모든 프로젝트에 공통으로 들어가는 자산이다.

- `settings.json`
- `hooks/db-read-only.sh`
- `hooks/vcs-no-commit.sh`
- `.codex/config.toml`
- `.gitignore`
- `commands/ehr/*.md.skel`

즉, 안전장치와 공통 실행 환경은 여기서 들어온다.

### 2. `profiles/ehr4` 또는 `profiles/ehr5`
프로젝트 기술 스택에 따라 갈리는 프로필별 자산이다.

- `skeleton/AGENTS.md.skel`
- `skeleton/CLAUDE.md.skel`
- `skeleton/README.md.skel`
- `agents/*.md`
- `skills/*/SKILL.md.skel`
- `reference/CODE_MAP.md`, `DB_MAP.md`

즉, 문서 뼈대와 역할별 스킬/에이전트 본문은 여기서 들어온다.

### 3. `skills/ehr-harness/lib`
겉으로는 잘 안 보이지만 생성 과정을 실제로 움직이는 분석/병합/검증 로직이다.

- `detect.sh`
- `analyze.sh`
- `audit.sh`
- `merge.sh`
- `harness-state.sh`
- `db.sh`
- `ehr-compound.sh`

신입 입장에서는 이 폴더를 "복붙 도구"가 아니라 "하네스 생성 엔진"이라고 보면 된다.

## 사용자가 "하네스 만들어줘"라고 했을 때 실제로 일어나는 일
원본 `SKILL.md`를 보면 적용 흐름은 거의 아래 순서로 고정되어 있다.

### 1. 플러그인 위치를 찾는다
- 먼저 플러그인 설치 경로를 찾는다.
- marketplace 설치인지, git clone 개발본인지 구분한다.

### 2. 의존성을 확인한다
- `superpowers` 플러그인이 없으면 생성 자체를 멈춘다.
- 즉, 원본 하네스는 단독 동작보다 상위 워크플로우와의 연계를 전제로 한다.

### 3. 현재 프로젝트가 어떤 상태인지 판정한다
프로젝트를 바로 덮어쓰지 않고 먼저 모드를 정한다.

- `fresh`: 하네스 흔적이 거의 없는 신규 프로젝트
- `legacy`: 하네스 흔적은 있지만 `.claude/HARNESS.json` 같은 스탬프가 없는 상태
- `stamped`: 이미 관리되는 하네스가 깔려 있는 상태
- `audit`: 사용자가 점검이나 drift 체크를 요청한 상태

여기서 중요한 점은, 원본 플러그인이 무조건 덮어쓰는 구조가 아니라는 것이다.

### 4. EHR4/EHR5 중 어떤 프로필인지 감지한다
원본 플러그인은 프로젝트 소스를 보고 프로필을 고른다.

- `pom.xml` + MyBatis 계열이면 `ehr5`
- `build.xml` + Anyframe 계열이면 `ehr4`

즉, 같은 하네스라도 프로젝트마다 들어가는 문서/스킬/에이전트가 달라진다.

### 5. 프로젝트를 깊게 분석한다
여기가 핵심이다. 단순 파일 복사가 아니라 아래 같은 실측값을 뽑는다.

- 모듈 맵
- 법칙별 사용 수
- 세션 변수 목록
- `authSqlID` 목록
- 재구현 금지 컴포넌트 참조 수
- 치명 프로시저 존재 여부
- 전체 프로시저/트리거 목록
- DB 접속 가능 여부
- Storybook 존재 여부

이 값들이 나중에 `AGENTS.md`, `CLAUDE.md`, `README.md`, 각종 `SKILL.md` 안에 치환되어 들어간다.

## 프로젝트 안에 실제로 무엇이 생기나
Step 3 기준으로 보면, 최종적으로 프로젝트에는 아래 자산이 심어진다.

### 루트 문서
- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- `GEMINI.md`

### Claude 전용 자산
- `.claude/settings.json`
- `.claude/HARNESS.json`
- `.claude/hooks/db-read-only.sh`
- `.claude/hooks/vcs-no-commit.sh`
- `.claude/agents/*.md`
- `.claude/skills/*/SKILL.md`
- `.claude/commands/ehr/*.md`

### Codex/Gemini 호환 자산
- `.agents/skills/*/SKILL.md`
- `.codex/config.toml`

### 기타
- `.gitignore`

이 구조를 보면, 하네스가 프로젝트에 입혀진다는 말은 결국 "프로젝트 루트에 AI 작업용 운영체제가 하나 들어온다"에 가깝다.

## 어떤 파일이 어떻게 만들어지나
생성 방식도 전부 같지 않다.

### 그대로 복사되는 것
- hooks
- 일부 agents
- 공통 설정 파일
- profile의 고정 reference

### skeleton에 실측값을 넣어서 렌더링되는 것
- `AGENTS.md`
- `CLAUDE.md`
- `README.md`
- 대부분의 `SKILL.md`

예를 들어 `AGENTS.md`에는 아래 같은 실측값이 들어간다.

- 시스템명
- 모듈 맵
- 세션 변수
- 치명 프로시저 목록
- 권한 모델
- DB 검증 전략
- 분석 스냅샷

즉, 같은 템플릿을 쓰더라도 프로젝트마다 완전히 같은 문서가 나오지 않는다.

## 안전장치는 어떻게 입혀지나
원본 플러그인의 공통 자산 중 가장 실무적인 부분이다.

### DB 읽기 전용 훅
`.claude/settings.json`이 `PreToolUse` 훅으로 `db-read-only.sh`를 등록한다.

이 훅은 DB 접근 명령에서 아래 계열 키워드를 차단한다.

- `DROP`
- `DELETE`
- `UPDATE`
- `INSERT`
- `MERGE`
- `ALTER`
- `CREATE`
- `EXEC`
- `CALL`

즉, AI가 DB를 함부로 바꾸지 못하게 기본 안전장치를 먼저 깔아준다.

### VCS 커밋 차단 훅
`vcs-no-commit.sh`도 함께 등록된다.

즉, 원본 하네스는 "AI는 코드 수정까지만, 실제 커밋/푸시는 사람이 판단"이라는 운영 철학을 같이 심는다.

## 업데이트 때는 어떻게 다르게 움직이나
이미 하네스가 깔린 프로젝트에서는 무조건 새로 생성하지 않는다.

### `legacy`
- 예전 흔적은 있는데 스탬프가 없으면 사용자에게 방향을 묻는다.
- adopt, merge adopt, 전체 재생성 중 하나를 고르게 한다.

### `stamped`
- 기존 하네스와 새 플러그인 자산의 diff를 계산한다.
- 안전한 항목, 새 항목, 충돌 항목을 나눠서 반영한다.

### `audit`
- 저장된 `HARNESS.json::analysis`와 현재 재분석 결과를 비교한다.
- drift를 계산한 뒤, 어떤 항목을 반영할지 정하게 한다.

신입 관점에서는 여기서 중요한 포인트가 하나 있다.

- 원본 플러그인은 "생성 도구"이면서 동시에 "유지보수 도구"다.

## 왜 `HARNESS.json`이 중요한가
`.claude/HARNESS.json`은 이 프로젝트 하네스의 상태표 같은 파일이다.

여기에 저장되는 것:

- schema version
- plugin version
- 선택된 profile
- 생성 시각
- source hash
- output hash
- analysis 결과
- `ehr_cycle` 누적 이력

즉, 다음 번 업데이트나 audit 때 이 파일이 기준점 역할을 한다.

## 신입 개발자가 이 단계에서 꼭 가져가야 할 이해
### 1. 하네스는 "문서 세트"가 아니다
분석 결과를 바탕으로 규칙·스킬·에이전트·안전장치를 프로젝트에 심는 패키지다.

### 2. `shared + profile + analysis` 세 축이 같이 움직인다
공통 자산만 봐도 안 되고, 프로필만 봐도 안 되고, 분석 로직만 봐도 안 된다.

### 3. 생성보다 유지보수 흐름이 더 중요할 수 있다
실무에서는 신규 생성보다 업데이트, merge, audit가 더 자주 필요할 가능성이 크다.

## 이 단계에서 확인할 체크포인트
- `shared` 자산과 프로필별 자산이 실제로 분리되어 있는가
- 어떤 파일이 복사되고 어떤 파일이 렌더링되는지 구분되는가
- `.claude/HARNESS.json`이 왜 필요한지 설명할 수 있는가
- 생성, 업데이트, audit 세 모드를 구분해서 말할 수 있는가
